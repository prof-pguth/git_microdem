DEMIX processing Steps

Collect Reference data

  • A 1” DEM cannot be compared directly with a 1-5 m UTM DEM, due to issues with both the projection and the spatial scale.  Almost all high resolution DEMs use a UTM or similar projection, and the global DEMs are all in geographic coordinates.  The global DEMs at 1” (approximately 30 m, but increasingly rectangular pixels as the latitude increases) show a generalized version of the terrain compared to 1-5 m DEMs.  While reinterpolation has been used to solve the projection issues, Guth and Kane (2021) showed that reinterpolation changed the characteristics of the DEM, and that care in the algorithms allows accurate computations with the geographic coordinates without introducing interpolation errors. Take the reference DEM or point cloud, and create the “best” 1” DEM
  • Match the pixel-is-area (ALOS) and pixel-is-point (5 other global DEMs) geometries (Guth and others, 2021) of the DEMs to be compared.  Because we will show ALOS is one of the winners of the wine contest, we do not want to have to interpolate some but not all of the DEMs and potentially bias the results.
  • Take the reference DEM and create the “best” possible 1” DEM which shows what a 1” global DEM should represent, preserving the original grid geometry of the DEM to be tested.  We take that as the mean elevation of the points in the reference high resolution DEM that map to the pixel in the 1” DEM using the appropriate pixel geometry.  If using a raw lidar point cloud, this should be the median to eliminate the effects of noise from isolated returns.
  • Match the spacing that changes with latitude for some of the 1” global DEMs (COP and ALOS), and which do not change in the same manner.
  • Ideally create both a DSM and a DTM, since only one of the global 1” DEMs being considered purports to be a DTM.  In fact all of the global DEMs probably fall somewhere between a true DTM and DSM (Guth and Geoffory, 2021).
  • Move the reference DEMs to the EGM2008 vertical datum if required, using the transformation grids from PROJ https://proj.org/) via OSGEO4W (https://www.osgeo.org/projects/osgeo4w/) or local geoids available from national/regional institutions.
  • We will have up to 4 reference DTMs for each DEMIX tile: 1”x1” pixel-is-point DTM, 1”x1” pixel-is-area, 1”x COP spacing pixel-is-point, and 1”x ALOS spacing pixel-is-area.  The last two will only be present for high latitude DEMs.  If we have a reference DSM, there will be 4 comparable reference DSMs.  This allows us to perform pixel by pixel comparisons for every pixel for difference criteria without having to interpolate in either the reference or candidate DEM.

 

  • Download reference data from national mapping agency or comparable source.  Place each area in a separate directory, with the DSMs and DTMs further separated.
  • Test areas must include at least one DEMIX tile with no more than 75% data voids.  A DEMIX covers covers about 10 x 10 km, but you must typically include a significantly larger area
    • Even when the source DEM tiles are 10x10 km on a UTM grid they will not include full DEMIX tiles.
    • DEM tiles frequently are not full becasue of
      • Coastlines
      • Mapping projects end at political boundaries
  • Test area naming conventions, with each high resolution DEM placed in its own directory
    • Separate words with "_"
    • If there is just a DTM, the name does not have to include anything.  DTM will be entered automatically in the reference DEM type field of the database.
    • If there is both a DTM and DSM, the last four characters in the file names  will be "_dsm" and "_dtm".  These are stripped off for the analysis, and DTM/DSM entered in the reference DEM type field of the database.
    • Create a DBF table with AREA, and VERT_CS.  The second is the EPSG code for the local vertical reference, which is 3855 if the DEM is already on EGM2008
  • Insure you know the vertical and horizontal datums of the DEMs, how to transform them.  MICRODEM will use GDAL/PROJ
    • Insure reference DEM is in WGS84 or equivalent.   
    • Insure reference DEM is in EGM2008.
  • Downsampling by aggregation  (creating a lower resolution grid), DEMIX 1 sec resample by averaging.  These will be floating point.

  • Create an index database for each test area.  Follow the format in the samples at https://doi.org/10.5281/zenodo.7600699/a>

    • .  You must have links to the vertical datum transformation if the files are not already EGM2008.

 

Processing Pipeline

  • We are currently not directly using lidar point clouds, but MICRODEM has the capability to use them to create reference DEMs.
    • For the DTM, use the median (to avoid the influence of rogue returns on the mean) of the points in the cloud classified as ground that map to each pixel in the DEM being created
    • For the DSM, use the median (to avoid the influence of rogue returns on the mean) of the first returns that map to each pixel in the DEM being created
  • Input reference DEM is in WGS84 or equivalent and EGM2008
  • Prep high resolution reference DEM to compare to 1" Global DEM 
  • (Optional) Keep test area DEM covering multiple DEMIX tiles, so operations like slope can work right to edge of tile, and to minimize the number of small files.  If you have multiple tiles in each area, this will be easier than trying to subset into DEMIX tiles.
  • Find DEMIX tiles in reference area DEM. 
  • Open and merge global DEM tiles to cover the test area.  This currently uses MICRODEMs Map library index.
  • Convert global DEMs to EGM2008 if necessary, at which point they become floating point.
  • Run statistics for reference DEM(s)
  • Find percentage of each DEMIX tile with valid reference data
  • For DEMIX tiles with > 75% reference data (potentially less than 100% because of coastalines, or the survey cut off before the DEMIX tile edge), run wine contest criteria currently coded in MICRODEM.
