Prepping a high resolution reference DEM to compare to a 1" Global DEM

  • Comparing point values in a high resolution DEM, which represent a very small area, with the values in the global DEM which represent a much larger area, has been done in a lot of studies, but is probably not ideal.  Instead we should compare the global DEM with what the true value should be of a DEM representing a 1" pixel, using the high resolution reference DEM to create a 1" reference DEM.
  • The 1" reference DEM has "comparable" elevations for every pixel, both in sampled area size and location.  Simple map algebra can create difference maps and statistics between the reference and candidate DEM.
  • For slopes and aspects (and to some extent elevations), we could either:
    • Use the 1" reference DEM, and compute slope and aspect from it.  In this case differences do not reflect the pixel size, but the DEM characteristics.  This is the preferred method.
    • Compute slope and aspect at every point in the reference DEM in its original resolution, and then take some kind of average slope and aspect of every point that is within each pixel of the 1" global DEM.  We would expect the reference DEMs to have steeper slopes due to their smaller pixel sizes.  If we have reference DEMs of different resolutions, we cannot compare test areas.

Steps:

  • Horizontal datum:
    • If needed convert high resolution reference DEM to WGS84
    • For 1" DEMs, conversion probably not needed for NAD83 or ETRS89 and perhaps some national datums
    • For 1" DEMs, probably no need for the realizations in time for the horizontal datum
  • Vertical datum
    • Most geotiffs do not have the vertical datum specified, and it takes some digging to find it (and then more digging to find how the local vertical datum relates to the WGS84 ellipsoid)
    • Need EGM96 for ALOS, ASTER, SRTM, and NASADEM, but we will shift all of those to EGM2008
    • Need EGM2008 for COP
    • Probably assumes WGS84 ellipsoid
    • Real challenge is to get conversion from a local datum (e.g. NAVD88 in North America, SLOVRP2016-Koper in Slovenia, or Rednap in Spain) to EGM
    • Use Geotiff or GTX grids.  For EGM96 in particular, these are relatively coarse resolution.  Use the files from PROJ
    • NGA has Fortran code and coefficients to do the conversions at any point (https://earth-info.nga.mil/index.php?dir=wgs84&action=wgs84).  These run from the command line, with an ASCII input and output files.
      • For EGM96, the coefficients file is only 5 MB, and 15' grids might be the best attainable with the model.  Interpolation within the 1' file is likely to be as good as a higher resolution model.
      • For EGM2008, the coefficients for the model are 236 MB, and the 1' grid is much more detailed that for EGM96.
    • MICRODEM can use the NGA programs to compute, using the full model resolution, EGM96 and EGM2008 geoid values at any desired point, but it is slow
  • Projection
    • Ultimately we want the reference DEM in geographic coordinates
    • For getting the 1" geographic reference DEM, reproject the original high resolution reference DEM to UTM or geographic if required (might be for MICRODEM, as other projections have not been tested since they are relatively rare)
  • DEMIX tile extent: does a 10 km tile, for the pixel-is-point DEMs, include the points on the bounding parallels and meridians (for pixel-is-area this is not a concern).  They will then be double counted, but I think that is not a problem.
  • Resampling the reference DEM to get a 1" version
    • Grid storage, the pixel representation to avoid 1/2 pixel posting shift.
      • Need pixel-is-point for COP, ASTER, SRTM, and NASADEM
      • Need pixel-is-area for ALOS
    • Lidar point cloud as the source of reference data
      • Choose DSM or DTM as desired result
      • Use any tool, providing the methodology can be defended
      • MICRODEM can create the grid using mean or median, and either the first returns, all returns, or just ground classified points.  This can create any desired grid spacing, with UTM, geographic, and some WKT projections with restrictions.  This is currently done in the GUI, but could easily be adapted to the command line.
        • 1 pass grids  Create a new grid that matches the DEM covering the map.  This can use min, max, or mean points in the pixel.
        • Multi pass grid: if you want the median, or a specified percentile.  This is slower, but avoids influence of rogue returns
    • DEM as the source of reference data
      • Most of the DEMs will be a DTM, which don't compare well with the 1" global DEMs which are DSMs
      • Use any tool, providing the methodology can be defended
      • MICRODEM can Average to new projection  which takes the mean of all points in the reference DEM that contribute to the 1" pixel.   MICRODEM can do this with either the GUI or the command line.  See example for El Hiero, with 2.5 m source DEM resampled to 1".
      • Take median of all points in the reference DEM that contribute to the 1" pixel.  This can be done using LAStools (demzip) to create a point cloud with the DEM, and then using the point cloud option above.

Using ICESat-2 to verify correct vertical datum

  • Converting to the same vertical datum (EGM2008) might be the hardest part of the comparisons, and it can be verify to determine the local vertical datum and its relation to EGM2008
  • ICESat-2 offers a way to check this, using only the highest quality returns
    • Get the ICESat-2 data
    • Prep the ICESat-2 data (in particular, convert the elevations to EGM2008)
    • Filter the data
      • Terrain elevation uncertainty above 0.1 m to remove points with no uncertainly
      • Terrain elevation uncertainty below 1 m
    • Interpolate the elevation from the reference DEM at the location of each IECSat-2 elevation.  Remove all ICESAT points where the DEM elevation is less than 0. 5m (these are water)
