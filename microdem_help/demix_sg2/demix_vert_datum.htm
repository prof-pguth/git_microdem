DEMIX Vertical Datum Shifts for Version 2 of DEMIX Database

 

The reference high resolution DEMs have several distinct reference systems.  Our goal is a reference DEM on the WGS84 ellipsoid and the EGM2008 geoid.

  • USGS 3DEP are on the NAD83 ellipsoid and NAVD88 geoid.  There is not a grid to convert, because you must convert the x and y coordinates a variable amount across the DEM, as well as the z coordinates.  For version 1 of the database we used the GEOID18 grid and ignored the differences in the horizontal datum.  For version 2 we first used US VDATUM to compute an average shift for each area, and confirmed using GDAL that the differences were "small", and now are using GDAL.
  • Spain, Norway:  on the WGS84 ellipsoid and a national geoid (two for Spain, mainland and Canary islands).  These only involve a shift in the vertical coordinates, and there are grids to accomplish this.  This requires two grids, one to move the elevation from the national geoid to the WGS84 elllipsoid, and a second grid to go from the ellipsoid to the EGM2008 geoid.  The EGM2008 grids are derived from NGA, and are included in the US VDATUM download, or OSGeo4W.  We are using mapalgebra with OSGEO4W national grids for Spain and Norway: OSGeo4W\share\proj\es_ign_egm08-rednap.tif, OSGeo4W\share\proj\es_ign_egm08-rednap-canarias.tif, OSGeo4W\share\proj\no_kv_HREF2018B_NN2000_EUREF89.tif
  • UK: we are using a map algebra and vertical transformation grid from OSGeo4W  OSGeo4W\share\proj\uk_os_OSGM15_GB.tif
  • Italy:  this uses a local geoid from the two provinces which produced the lidar DEMs, which supply a transformation grid which doe not have a ESPG code.  We used map algebra.  Without an EPSG code we cannot use GDAL.
  • South America: these were converted by co-authors using unspecified methodology.
Program Data Notes
GDAL Grid Need EPSG codes for both ellipsoids and vertical datums.
OSGeo4W installation will install the transformation data, which are used by PROJ and GDAL.
Slow when both horizontal and vertical datums have to shift.
Will create a new grid with an offset corner location, and the same number of rows and columns.  It is unclear if the horizontal shift is exact for the corner, or an average for the entire grid.
This agrees with very small differences with what ArcGIS does (check by a colleague).
Simple map algebra in GIS Grid Need two conversion grids, one between the local geoid and the ellipsoid, and another from the ellispoid to EGM2008 from NGA, and do a subtraction and addition with the DEM. OSGeo4W installation will install these, which are used by PROJ and GDAL.
This will not handle a horizontal datum (ellipsoid shift)
