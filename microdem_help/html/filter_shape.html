Square versus Round Filters

Filters can be square or "round", and figured in terms of either pixel or meters.  Meters will be more appropriate when considering the arc second DEMs.  We will use filter interchangeably with kernel or matrix.

Because the matrix with the filter will always  be centered on the pixel and be symmetrical about that point, the size will be odd.

Filters are defined either in terms of the radius, or the size of the filter matrix.  A radius 2 filter will have a 5x5 matrix. 

A square filter will have a value for every element in the matrix, such as 3x3, 5x5, 7x7.  It could:

  • Use the points at each location, ignore the different horizontal distances.
  • Linearly interpolate values at the same distance within the grid
  • Find the elevation at the same distance, assuming a constant slope between the center point and the nearest point to the desired location

A round filter will not compute values that are too far from the center pixel, which will be in each corner.  These will have s value of 0 or some equivalent depending on how the coding is implemented.

  • The size of the filter could be based on either based on the pixel size, or the diagonal.  This makes a big difference for the 3x3 filter in the case below.  If the filter size is 30 m (1 pixel), only the four nearest neighbors will be considered.  If the filter size is 45 m, all 8 nearest neighbors will be considered.  We think that for the case of the 3x3 filter using all 8 neighbors makes more sense.
  • For the values on the diagonals, you have the same choices as above.

The units of the result are import. 

  • For slope, the units are essentially dimensionless, and do not depend on the difference over which they are measured, because the vertical difference is divided by the horizontal difference.  The common options for slope are percent, degrees, or m/m, all of which are dimensionless.  There are slight differences because slope decreases with increasing region size.
  • Other measures, like relief or TPI, consider only the vertical differnces, and will increase with increasing measurement area.

An operation like TPI takes the 8 points surrounding the central point, and averages the differences in elevation from the central pixel.  It does not consider the different distances to the diagonal points.  This is essentially a relief computation.  An improvement to the algorithm is to normalize the distances.  MICRODEM takes the NS distance, and for the diagonal, finds the elevation that would occur at that distance using the slope between the central point and the diagonal neighbor.  This elevation will be less than the elevation at the point on the diagonal.

For the arc second DEM, the spacing in x and y differ.  This means that a filter defined in meters might effectively be rectangular, with a different number of rows and columns.

For normalizing the distances for TPI, both the diagonals and the EW points must be adjusted.  The EW points used will be higher, because the EW distances are smaller and the points corresponding to the NS distances are farther away.

The Delphi language does not have built in matrix functions, which colors how I approach the issue.

  • The matrix multiplcation implementation in MICRODEM may or may not be particularly fast.
  • Even if it is not fast, the number of computations may mean the speed loss is not significant.
  • As an alternative, the algorithm can use the linear interpolations it already does to get elevations to get the elevations required.
  • The direction linear interpolation would become increasingly attractive as the size of the kernel increased and the number of 0 elements increased.  These require multiplications and then additions for every point in the matrix, and the direct interpolation for the required points would require significantly fewer.
  •  For UTM grids, the kernels can be computed once and stored.  For geographic grids, they probably have to be computed for every latitude, making the direct computations of desired, interpolated points more attractive.
